# Observability（可観測性設計）

```
種類: Generator
入力: 設計 型の成果物
出力: 運用 型の変種（docs/<機能名>/ops/observability.md、status ヘッダ付き）
完成基準: 下記
```

## 出力の型

observability.md（frontmatter: `type: 運用` / `feature:` / `status:`）に:

1. **ログ設計** — | どこで | 何を | レベル | 含めるもの / 含めないもの |
   - 必須: リクエスト境界（入口と出口）、エラー、外部呼び出しの失敗
   - 相関 ID の方針（1リクエストを追跡できるか）
2. **メトリクス** — サービスなら RED（リクエスト数 / エラー率 / 処理時間）、リソースなら USE（使用率 / 飽和 / エラー）を基本セットに:
   | メトリクス | 何が分かる | 集め方 |
3. **アラート** — | 条件（閾値） | 深刻度 | 鳴ったら何をする（runbook の該当節へリンク） |
4. **ダッシュボード** — 障害時に最初に開く1枚に何を並べるか

## ルール

- **「鳴ったら何をするか」の無いアラートは作らない。** 対応のないアラートはただの騒音になり、本物を無視させる
- **ログに秘密情報を書かない。** パスワード・トークン・個人情報はログ設計の「含めないもの」に明記する
- **NFR の数値と接続する。** nfr.md に「p95 500ms」とあるなら、それを測るメトリクスとアラートがここに無ければならない
- **少なく始める。** メトリクスは基本セット＋ドメイン固有の2〜3個から。使わないダッシュボードを量産しない

## 完成基準

- [ ] エラーが起きたとき、ログだけで「どのリクエストの・どこで・何が」を特定できる設計になっている
- [ ] すべてのアラートに対応アクション（runbook 参照）がある
- [ ] NFR の数値要件がメトリクスで観測できる
- [ ] status ヘッダを付けた
