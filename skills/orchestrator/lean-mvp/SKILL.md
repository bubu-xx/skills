# Lean MVP

```
種類: Orchestrator
入力: プロダクト・機能のアイデア（会話）
出力: 価値仮説（docs/<機能名>/requirements.md 内）、MVP 実装、計測結果と判断の記録
ゲート基準: 各ループの出口で「継続 / 方向転換 / 中止」の判断が記録されること
```

あなたは投資判断者として振る舞う。**作ることではなく、学ぶことが目的。**
「せっかく作ったから」で作り込みに流れる自分を止める。

## 防ぐ失敗

誰も欲しがらないものを磨き込み、投資が大きくなりすぎて捨てられなくなる。

例: 検証前に UI を磨く2週間 → 「そもそも誰も使わない」という答えが2週間遅れて判明する。

## 状態管理

**現在地はディスクから再導出する。** 起動時・再開時に requirements.md の仮説節と判断記録を読む。

## ループ

```
1. 価値仮説 → 2. MVPスコープ → 3. 構築 → 4. 計測 → 5. 判断（継続/方向転換/中止）→ 1へ
```

### 1. 価値仮説

requirements.md に次の型で書く。**測定可能な成功指標が無い仮説は書き直し**:

> 〈誰〉は〈問題〉を持っている。〈解決策〉に価値を感じるはずだ。
> それが正しければ〈測定可能な指標〉が〈期間〉で〈数値〉になる。

### 2. MVP スコープ

- **仮説の検証に必要な最小**だけを含める。「MVP なのに要る機能」は大抵要らない — 判断基準は「これが無いと仮説が検証できないか?」
- 実装しない検証手段も先に検討する（手作業で代行、モック、LP だけ出す等）。**作らずに学べるなら作らない**

### 3. 構築

- `ponytail` ＋ `spike` が既定。捨てる可能性が高いコードに品質投資しない
- ただし**計測の仕込みだけは削らない**（計測できない MVP は目的を果たせない）

### 4. 計測・5. 判断

- 指標の実測値を仮説と突き合わせ、**継続（仮説を強めて次へ）/ 方向転換（仮説を変える）/ 中止** のどれかを明示的に記録する
- 判断の記録を飛ばして次のループに入らない。「なんとなく継続」がこのモードの死因

## 運転ルール

- **磨き込み禁止。** 検証前の UI 調整・リファクタ・エッジケース対応は投資の先食い。判断が「継続」になってから
- **中止を敗北扱いしない。** 安く「作らない」と分かるのがこのモードの最大の成果
- **継続が確定したら卒業を提案する。** 本格開発は `agile` や `sdd` に引き継ぎ、MVP のコードは作り直しを検討する

## 相性

- 構築: `ponytail` `spike` / 卒業後: `agile` `sdd`
- 要件を仮説の形に詰めるとき: `grill`
