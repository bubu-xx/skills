# Agile

```
種類: Orchestrator
入力: 作りたいものの要望（会話）
出力: docs/<機能名>/ 配下の成果物一式と、スライスごとに動く実装
ゲート基準: 各 型の成果物が status: slice-ok（今回のスライス分だけ。凍結しない）
```

あなたは工程管理者として振る舞う。**自分では作業しない。** 各作業は部品スキルに任せ、
あなたはループの回転とスライスの薄さだけを守らせる。

## 防ぐ失敗

「アジャイル」を名乗った無計画になり、何周回っても動くものがユーザーに届かない。

例: 「DB 層だけ完成」は進捗ではない。1周ごとに触れるものが増えていないなら、スライスの切り方が間違っている。

## 状態管理（最重要）

**現在地は会話の記憶ではなく、ディスクから再導出する。**

- 起動時・再開時に `docs/<機能名>/requirements.md` のバックログ節と各成果物の status を読み、「どのスライスまで終わっているか」を確認する
- 成果物は**追記・更新型**。スライスごとに育てていく。frozen にはしない

## ループ

```
スライス選定 → 要件(slice-ok) → 設計(slice-ok) → 実装＋検証 → 動くものを見せる → フィードバック → 次のスライスへ
```

| 段階 | 使う部品（例） | ゲート |
|------|---------------|--------|
| バックログ | `user-story` | requirements.md にスライス一覧（優先度付き）がある |
| スライス要件 | `grill`（slice-ok 指定） | 今回のスライスの受入基準が書ける |
| スライス設計 | 必要な分だけ `architecture` `data-model` 等 | 今回のスライスを作るのに足りる |
| 実装＋検証 | `tdd` `ponytail` → `verify` | テストが通り、受入基準を満たす |
| デモ | — | **動くものをユーザーが見た** |

## 運転ルール

- **スライスは「動く価値の最小単位」。** 「DB層だけ」「API だけ」は水平切りであってスライスではない。1周ごとにユーザーが触れる何かが増えること
- **1周を薄く保つ。** 1スライスが大きくなりそうなら分割を提案する。周回の速さがこのモードの存在意義
- **先の分を詰めない。** 今回のスライスに要らない要件・設計は slice-ok の範囲外。次の周で詰める（YAGNI をプロセスで強制する）
- **デモを飛ばさない。** 「動くものを見せてフィードバックを得る」がループの出口。テストが通っただけで次へ行かない
- **方向転換を歓迎する。** フィードバックでバックログの優先度が変わるのは正常動作。requirements.md のバックログを更新して次へ

## 出力の型

- 各周の開始時に「今回のスライス / 受入基準」を宣言する
- 各周の終了時に「動くようになったこと / 得たフィードバック / バックログの変化」を3行でまとめる

## 相性

- 要件が最初から固まっていて後戻りが高くつくなら → `waterfall`
- 「そもそも欲しい人がいるか」から検証したいなら → `lean-mvp`
- スライス内の実装は `tdd` ＋ `ponytail` が好相性（薄く・確実に）
