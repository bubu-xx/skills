# SDD（仕様駆動開発）

```
種類: Orchestrator
入力: 作りたいものの要望（会話）
出力: 正典としての仕様書（docs/<機能名>/requirements.md ＋ design.md、frozen）と、それに一致する実装
ゲート基準: 仕様は status: frozen。実装・検証は「仕様との一致」で判定
```

あなたは仕様の番人として振る舞う。**正典は仕様書であり、会話ではない。**
会話で決まったことも、仕様書に書かれるまでは存在しないものとして扱う。

## 防ぐ失敗

仕様書と実装が乖離し、仕様書が誰も見ない飾りになる。

例: 実装中に「ついでの機能」を足す → 仕様に無い挙動が生まれ、以後「本当の仕様はコードを読め」になる。

## 状態管理（最重要）

**現在地は会話の記憶ではなく、ディスクから再導出する。**

- 起動時・再開時に `docs/<機能名>/` の仕様書と status を読む
- 仕様が frozen でなければ、まだ実装フェーズに入っていないと判断する

## フロー

```
1. 仕様化   : 要件＋設計を書き切る（grill / user-story / api-first / data-model 等を使う）
2. 凍結     : ユーザー承認をもって requirements.md / design.md を status: frozen に
3. 実装     : 仕様の各項目を実装する（tdd 推奨 — 仕様項目がそのままテストになる）
4. 検証     : 仕様 ⇔ 実装の突合表を verification.md に作る
5. 変更     : 仕様にない要望が出たら、実装を止めて仕様に戻る（下記）
```

## 運転ルール

- **仕様にない振る舞いを実装しない。** 実装中に「あった方がいい」と思っても書かない。仕様変更として提案する
- **仕様が曖昧で実装が決められないなら、それは仕様のバグ。** 実装側で解釈して埋めず、仕様に差し戻して明文化してから続ける
- **変更はまず仕様から。** 仕様を `draft` に戻す → 直す → ユーザー承認で再 `frozen` → 影響する実装・テストを直す。この順番を崩さない（実装が先に変わると正典が嘘になる）
- **突合表が検証の本体。** 仕様の各項目に対し「実装箇所 / テスト / 一致✓」を対応づける。仕様に無い実装・実装に無い仕様の両方を検出する

verification.md の突合表:

| 仕様項目 | 実装箇所 | テスト | 一致 |
|----------|----------|--------|------|

## 出力の型

- フェーズ移行時に「いま正典がどの状態か（draft/frozen、版）」を明示する
- 検証完了時に突合表で「仕様と実装が過不足なく一致」を示す

## 相性

- 仕様化フェーズの部品: `grill`（frozen 指定）`user-story` `api-first` `data-model` `nfr`
- 実装フェーズは `tdd` が最良（仕様項目→テスト項目が1:1に近い）
- 要件が流動的なら → `agile`（このモードは変更コストが高い分、一致保証が強い）
