# Test Plan

```
種類: Generator
入力: 要件 型・設計 型の成果物
出力: 検証 型の変種（docs/<機能名>/test-plan.md、status ヘッダ付き）
完成基準: 下記
```

## 出力の型

test-plan.md（frontmatter: `type: 検証` / `feature:` / `status:`）に:

1. **対象と範囲** — 何をテストし、何をテストしないか（しない理由付き）
2. **観点×対象マトリクス**:

| 対象（機能/ユースケース） | 正常系 | 異常系 | 境界値 | 並行/順序 | 性能 |
|--------------------------|--------|--------|--------|-----------|------|
| ログイン | ✓ | ✓ | ✓ | — | — |

3. **テストケース一覧** — マトリクスの✓を具体化。| # | 対象 | 観点 | 入力/手順 | 期待結果 | 自動/手動 |
4. **実行環境** — どこで・何を使って回すか（ローカル/CI、フレームワーク）

## ルール

- **異常系・境界値から埋める。** 正常系は放っておいても書かれる。抜けるのはいつも異常系
- **期待結果が書けないケースは要件の穴。** test-plan 中に発覚したら `grill` に差し戻す（テスト計画は要件の検算でもある）
- **自動化できるものは自動に。** 手動テストは「自動化コストが見合わない理由」を書けるものだけ
- **「テストしない」も明記する。** 範囲外を書かない計画は、抜けと区別がつかない

## 完成基準

- [ ] 要件の各ユースケースがマトリクスの行にある
- [ ] すべてのテストケースに期待結果がある
- [ ] 自動/手動の別が付いている
- [ ] status ヘッダを付けた
