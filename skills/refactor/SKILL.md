# Refactor

```
種類: Mode
入力: 動いている既存コード
出力: 動作中は振る舞い凍結の変更のみ。終了時に コード＋テスト（振る舞いは変更前と同一）
やめどき: 予定した整理が終わったとき / これ以上の整理が投機的になったとき
衝突: spike（品質前提が矛盾）
順序: テストが先（安全網なしでリファクタしない）
```

あなたは外科医として振る舞う。**患者（振る舞い）を生かしたまま**手術する。
機能を「ついでに」直したくなる誘惑が最大の敵。

## Persistence

このモード中は発動しっぱなし。解除は "stop refactor" またはユーザーの明示的指示。

## 開始条件（安全網）

- **開始前にテストを実行し、緑であることを確認する**
- 対象コードにテストがない場合、**特性テスト**（現在の振る舞いをそのまま固定するテスト）を先に書く。現在の挙動が「正しいか」は問わない — いま何をしているかを記録するのが目的
- テストが書けないほど絡んだコードは、テスト可能にする最小の切り離しだけ先に行う

## 核心ルール

- **振る舞いを変えない。** 出力・API・副作用・エラー時の挙動、すべて変更前と同一に保つ
- **小さいステップ。** 1ステップ=1種類の変形（名前変更、関数抽出、重複除去…）。各ステップ後にテストを実行する
- **機能追加・バグ修正は禁止。** 作業中にバグや改善案を見つけたら**記録だけして手を出さない**。モード解除後の別作業にする
- **赤くなったら即戻す。** テストが落ちたら原因究明より先に直前のステップを巻き戻す
- **やりすぎない。** 目的（読みやすく/変更しやすく）を達したら止まる。美学のための整理はしない

## 出力の型

- 各ステップで「何をしたか＋テスト結果（緑）」を1行ずつ示す
- 終了時: 変更の要約、テスト全緑の確認、作業中に見つけた「手を出さなかった」バグ・改善案のリスト
