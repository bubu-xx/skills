# Threat Model（脅威モデリング）

```
種類: Generator
入力: 設計 型の成果物（構成が決まっていること）
出力: 設計 型の変種（docs/<機能名>/threat-model.md、status ヘッダ付き）
完成基準: 下記
```

## 防ぐ失敗

設計時なら1行で済んだ対策が、リリース後の発覚で全面改修＋インシデント対応になる。

例: ファイル名をそのままパスに使う設計 → 設計時なら「許可リスト検証を入れる」の1行。運用後の path traversal 発覚なら緊急対応と影響調査。

## 出力の型

threat-model.md（frontmatter: `type: 設計` / `feature:` / `status:`）に:

1. **守る資産** — 漏れたら/壊れたら困るもの（ユーザーデータ、認証情報、決済…）
2. **信頼境界と入口** — 外部から入力が入る場所すべて（UI、API、ファイル、Webhook、環境変数…）
3. **脅威表** — 入口×脅威分類で洗い出す:

| 入口 | 脅威 | 分類 | 対策 | 状態 |
|------|------|------|------|------|
| ログインAPI | 総当たり | なりすまし | レート制限 | 対策済/未 |

分類は STRIDE を簡略化した5つで見る: **なりすまし / 改竄 / 情報漏洩 / サービス妨害 / 権限昇格**

4. **受け入れる残余リスク** — 対策しないと決めたものと、その理由

## ルール

- **入口を全部列挙してから脅威に入る。** 入口の見落としは脅威の見落としに直結する
- **対策は具体的に。** 「適切にバリデーションする」は対策ではない。「サーバ側で長さ・型・許可リスト検証」が対策
- **「対策しない」も決定として書く。** 脅威表に載せた上で残余リスクへ。黙って落とすのが一番危ない
- **既存の仕組みを先に使う。** フレームワークの CSRF 対策・認証ライブラリで済むものを自作しない

## 完成基準

- [ ] すべての入口が脅威表に現れている
- [ ] すべての脅威が「対策あり」か「残余リスクとして受容」のどちらかになっている
- [ ] status ヘッダを付けた
