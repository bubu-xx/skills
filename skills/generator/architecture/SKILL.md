# Architecture

```
種類: Generator
入力: 要件 型の成果物
出力: 設計 型の変種（docs/<機能名>/design.md、status ヘッダ付き）
完成基準: 下記
```

## 防ぐ失敗

要件に根拠のない構成（10人向けツールにマイクロサービス等）が採用され、複雑さだけが残る。

例: 却下案の記録がない技術選定は、半年後に「なぜこれなの」から同じ調査をやり直すことになる。

## 出力の型

design.md（frontmatter: `type: 設計` / `feature:` / `status:`）に:

1. **構成図** — コンポーネントとデータの流れ（mermaid 推奨）
2. **コンポーネント表** — | 名前 | 責務(1行) | 依存先 |
3. **技術選定表** — | 領域 | 採用 | 却下した選択肢 | 理由 |
4. **データフロー** — 主要ユースケース1〜3個について、入力→処理→出力の経路を1行ずつ

## ルール

- **要件にない規模を設計しない。** 想定ユーザー10人のツールにマイクロサービスを書かない。要件（特に非機能要件）が根拠にない構成要素は削る
- **却下案と理由を必ず残す。** 「なぜ他でないのか」が後から一番聞かれる。大きな判断は `adr` に切り出してよい
- **境界を先、中身は後。** コンポーネント内部の詳細設計はここではしない
- **既存資産を先に調べる。** 既にあるインフラ・ライブラリ・パターンで済む部分に新技術を足さない

## 完成基準

- [ ] 構成図があり、要件の主要ユースケースが図の上でたどれる
- [ ] すべてのコンポーネントに1行の責務が書けている（書けない=分割が曖昧）
- [ ] 技術選定に却下案と理由がある
- [ ] status ヘッダを付けた
