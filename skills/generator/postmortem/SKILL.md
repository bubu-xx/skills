# Postmortem（ポストモーテム）

```
種類: Generator
入力: 障害の記録（会話・ログ・チケット）
出力: 運用 型の変種（docs/ops/postmortem/YYYY-MM-DD-<短いタイトル>.md）
完成基準: 下記
```

## 防ぐ失敗

振り返りが「気をつけます」で終わり、3ヶ月後に同じ障害が再発する。

例: ダメ「デプロイ時の確認を徹底する」。良い「マイグレーション未適用ならデプロイが自動で止まる CI チェックを追加する」。

## 出力の型

1障害1ファイル:

```markdown
# YYYY-MM-DD: <障害のタイトル>

## 要約
何が起きて、誰にどれだけ影響し、どう収束したか（3行以内）。

## 影響
- 期間: 検知 YYYY-MM-DD HH:MM 〜 復旧 HH:MM
- 影響範囲: <誰の・何が・どの程度>

## タイムライン
- HH:MM 何が起きた / 誰が何をした（事実だけ。解釈を混ぜない）

## 根本原因
直接原因から「なぜ」を繰り返して掘る。
「担当者がミスした」で止めない — ミスが事故になった仕組みまで掘る。

## うまくいったこと / いかなかったこと
- 検知は速かったか、手順はあったか、ロールバックは効いたか

## 再発防止アクション
| # | アクション | 期限 | 状態 |
```

## ルール

- **blameless。** 個人名を原因欄に書かない。「人がミスをしても事故にならない仕組みの欠如」が根本原因
- **事実と解釈を分ける。** タイムラインは事実のみ。解釈は根本原因の節で
- **アクションは仕組みに向ける。** 「気をつける」「周知する」は再発防止ではない。ガードレール（テスト、制約、アラート、自動化）に落とす
- **アクションゼロも許容する。** 対策コストがリスクに見合わないなら「受容する」と書いて終える。形だけのアクションを並べない

## 完成基準

- [ ] タイムラインが事実だけで書かれている
- [ ] 根本原因が個人ではなく仕組みを指している
- [ ] 各アクションに期限があり、仕組みの変更になっている（または受容の明記）
