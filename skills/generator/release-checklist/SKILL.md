# Release Checklist

```
種類: Generator
入力: 検証 型の成果物（検証が終わっていること）
出力: 検証 型の変種（docs/<機能名>/release-checklist.md）
完成基準: 全項目が ✓ か「対象外(理由)」になり、go/no-go が判定されたとき
```

## 防ぐ失敗

ロールバック手順なしでデプロイし、障害時に「戻せない」ことが判明して手詰まりになる。

例: 「戻せます」（未検証）→ 実際はマイグレーションが不可逆でデータごと戻せなかった。事前に一度試していれば分かったこと。

## 出力の型

release-checklist.md に、確認済み ✓ / 未 / 対象外(理由) を付けたリスト:

### 品質
- [ ] 全テストが通っている（CI の結果へのリンク or 実行ログ）
- [ ] 検証成果物（verification.md）が基準 status に達している
- [ ] 既知の未修正バグの一覧があり、リリース可否を判断済み

### 戻せるか（最重要）
- [ ] ロールバック手順が書いてあり、**実際に試した**
- [ ] データのバックアップが取れている / マイグレーションに戻し方がある
- [ ] 段階的リリースの仕組み（フラグ・カナリア）を使うか判断した

### 観測
- [ ] リリース直後に見るダッシュボード/ログが決まっている
- [ ] 失敗と判断する基準と、判断するタイミングが決まっている

### 周知
- [ ] 影響を受ける人への告知が済んでいる（ユーザー・チーム・運用者）
- [ ] リリースノート/変更履歴を書いた

### 後始末
- [ ] リリース後に消すもの（フラグ、旧コード、一時的な措置）が記録されている

## ルール

- **✓は検証してから付ける。** 「たぶん大丈夫」で✓を付けるならリストの意味がない。ロールバックは特に、書いてあるだけで試してないものが本番で一番裏切る
- **対象外には理由を書く。** 個人プロジェクトに告知は要らない — と書いて飛ばすのは OK、黙って飛ばすのは NG
- **項目はプロジェクトに合わせて増減してよい。** ただし「戻せるか」の節は削らない

## 完成基準

- [ ] 全項目が ✓ または 対象外(理由)
- [ ] go / no-go の判定が書いてある
