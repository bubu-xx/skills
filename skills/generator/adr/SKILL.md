# ADR（アーキテクチャ決定記録）

```
種類: Generator
入力: 直前に行われた（or これから行う）設計上の判断
出力: 設計 型の変種（docs/adr/NNNN-<短いタイトル>.md）
完成基準: 下記
```

## 防ぐ失敗

判断の理由が失われ、半年後に同じ議論と同じ調査が再燃する。

例: 「なんで MongoDB なんだっけ」→ 誰も覚えていない → RDB 移行の議論が振り出しから始まる。ADR が1枚あれば5分で終わる話。

## 出力の型

`docs/adr/` に連番で1判断1ファイル（例: `0003-use-postgresql.md`）:

```markdown
# NNNN: <判断のタイトル>

- 状態: 承認 | 提案中 | 廃止（→ NNNN で置換）
- 日付: YYYY-MM-DD

## 文脈
なぜこの判断が必要になったか。制約・前提。

## 決定
何をすることにしたか（1〜3行、断定形で）。

## 却下した選択肢
- <選択肢A> — 却下理由
- <選択肢B> — 却下理由

## 結果
この決定で受け入れたトレードオフ。良くなること・悪くなること・縛られること。
```

## ルール

- **1判断1ファイル。** 複数の決定を1つの ADR に詰めない
- **却下案の無い ADR は書き直す。** 選択肢が1つしかなかったなら、それは判断ではなく成り行き。記録する価値があるか疑う
- **記録に値する判断だけ書く。** 基準: 覆すのが高くつく判断（DB、フレームワーク、通信方式、データ形式）。変数名やディレクトリ構成は書かない
- **過去の ADR は書き換えない。** 判断が変わったら新しい ADR を足して古い方を「廃止」にする。履歴が資産

## 完成基準

- [ ] 文脈・決定・却下案・結果の4節が埋まっている
- [ ] 半年後の他人（未来の自分）が「なぜこうしたか」に答えられる内容になっている
