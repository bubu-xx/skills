# NFR（非機能要件）

```
種類: Generator
入力: 要件 型の成果物
出力: 設計 型の変種（docs/<機能名>/nfr.md、status ヘッダ付き）
完成基準: 下記
```

## 防ぐ失敗

「速ければいい」のまま出荷し、負荷で落ちて初めて本当の要件が判明する。

例: ダメ「高速であること」。良い「検索は p95 で 500ms 以内（負荷テストで計測）」。前者は設計を導けないが、後者はキャッシュ要否まで決める。

## 出力の型

nfr.md（frontmatter: `type: 設計` / `feature:` / `status:`）に、カテゴリごとの表:

| カテゴリ | 要件 | 数値・基準 | 測り方 |
|----------|------|-----------|--------|
| 性能 | 検索の応答時間 | p95 で 500ms 以内 | 負荷テストで計測 |

カテゴリの標準セット（該当しないものは「対象外」と明記して飛ばす）:

1. **性能** — 応答時間、スループット、同時接続数
2. **キャパシティ** — データ量の桁、ユーザー数、成長率
3. **可用性** — 稼働率、計画停止の可否、障害時の復旧目標（RTO）と許容データ損失（RPO）
4. **セキュリティ** — 認証方式、権限モデル、扱う機微情報とその保護
5. **運用性** — バックアップ、監視、ログ保持期間
6. **保守性・拡張性** — 想定される変更、対応すべき環境

## ルール

- **測定できない要件は書き直す。** 「高速であること」は要件ではない。「p95 で 500ms」が要件
- **全カテゴリを埋めない。** 個人ツールに稼働率 99.99% は要らない。「対象外」の明記も立派な決定
- **数値はユーザーに決めさせる。** こちらから相場を提示してよいが、確定は本人。ここの数字がアーキテクチャのコストを決める
- **「対象外」と「未検討」を区別する。** 空欄を残さない

## 完成基準

- [ ] 全カテゴリが「数値あり」か「対象外」のどちらかになっている
- [ ] すべての数値に測り方が書いてある
- [ ] status ヘッダを付けた
