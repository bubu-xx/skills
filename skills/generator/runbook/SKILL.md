# Runbook（運用手順書）

```
種類: Generator
入力: コード＋テスト、設計 型の成果物
出力: 運用 型の変種（docs/<機能名>/ops/runbook.md または docs/ops/runbook.md）
完成基準: 下記
```

## 防ぐ失敗

深夜の障害でソースコードを読みながら対応する羽目になり、復旧が数時間遅れる。

例: 「DB 接続エラーの症状 → この3コマンドで確認 → この手順で復旧」が書いてあれば5分。無ければ調査から始まる。

## 出力の型

runbook.md に:

1. **システム概要** — 何が・どこで動いているか（1画面に収まる分量）
2. **定常操作** — 起動 / 停止 / 再起動 / デプロイ / ロールバック。それぞれ**コピペで実行できるコマンド**で
3. **障害対応** — 症状ベースで引けるように:

```markdown
### 症状: <ユーザーから見えた現象>
1. 確認: <このコマンド/画面で状態を見る>
2. 原因の切り分け: <AならX、BならY>
3. 対処: <コマンド>
4. 収まらない場合: <エスカレーション先 / 次の手>
```

4. **定期作業** — バックアップ確認、証明書更新、ログローテーション等（あれば）
5. **場所一覧** — ログ、設定、秘密情報（の在り処への参照、値は書かない）、監視ダッシュボード

## ルール

- **深夜の自分が読む前提で書く。** 判断を要求しない。「適宜」「必要に応じて」を使ったら書き直す
- **コマンドはコピペ可能な完成形で。** `<host>` のようなプレースホルダは冒頭に実値の一覧を置いて解決する
- **症状から引けるようにする。** 障害時は原因ではなく症状しか見えていない
- **秘密情報は書かない。** 在り処（シークレットマネージャのキー名等）だけ書く
- **手順は一度実際に流して検証する。** 動かない手順書は無いより悪い

## 完成基準

- [ ] 定常操作がすべてコピペ実行可能
- [ ] 想定される主要な障害症状が最低3つ載っている（思いつかないなら `adversarial-review` の結果を使う）
- [ ] 手順を一度実際に流した
