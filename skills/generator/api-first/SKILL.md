# API First

```
種類: Generator
入力: 要件 型の成果物
出力: 設計 型の変種（docs/<機能名>/api.md、status ヘッダ付き。プロジェクトが OpenAPI を使うならそちらの形式）
完成基準: 下記
```

## 防ぐ失敗

フロントとバックが別々の想像で実装し、結合の日に形式が合わず崩壊する。

例: 「ユーザー情報を返す」だけの合意 → 日付が ISO8601 か UNIX 秒か、null か省略か、で結合日にもめる。実例 JSON が先にあれば起きない。

## 出力の型

api.md（frontmatter: `type: 設計` / `feature:` / `status:`）に:

1. **エンドポイント一覧** — | メソッド | パス | 概要 | 認証要否 |
2. **各エンドポイントの契約** — リクエスト例とレスポンス例（実際の JSON で。型だけの抽象記述にしない）
3. **エラー形式** — 全エンドポイント共通のエラーレスポンス構造と、主要エラーケース（400/401/404/409/500 等）の発生条件
4. **互換性方針** — 破壊的変更をどう扱うか1〜3行（バージョニング or 追加のみ等）

プロジェクトに既に OpenAPI / GraphQL スキーマがあるなら、その形式・その置き場所に従う。

## ルール

- **例で書く。** 「User オブジェクトを返す」でなく実際のレスポンス JSON を書く。曖昧さは例が潰す
- **エラーを先に決める。** 正常系だけの API 設計は半分しか終わっていない。呼ぶ側はエラー処理を書く必要がある
- **要件にないエンドポイントを作らない。** 「あとで使うかも」の CRUD 一式は書かない
- **命名は既存 API に合わせる。** プロジェクトに既存 API があるなら、その規約（複数形/ケース/ページング方式）を踏襲する

## 完成基準

- [ ] 要件の各ユースケースが、どのエンドポイント（の列）で実現されるか対応づく
- [ ] 全エンドポイントにリクエスト/レスポンスの実例がある
- [ ] エラー形式が共通化されている
- [ ] status ヘッダを付けた
