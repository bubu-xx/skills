# Data Model

```
種類: Generator
入力: 要件 型の成果物
出力: 設計 型の変種（docs/<機能名>/data-model.md、status ヘッダ付き）
完成基準: 下記
```

## 防ぐ失敗

整合性チェックがアプリ側だけになり、バグがゴミデータを作り、修復がテーブル設計の10倍高くつく。

例: メールの一意性をアプリでだけチェック → 同時登録で重複発生。UNIQUE 制約が1行あれば防げた。

## 出力の型

data-model.md（frontmatter: `type: 設計` / `feature:` / `status:`）に:

1. **ER 図** — mermaid の `erDiagram` で
2. **テーブル定義** — テーブルごとに | カラム | 型 | 制約 | 備考 |
3. **制約の一覧** — 一意性・参照整合性・NOT NULL 等、**DB 側で強制するもの**
4. **迷った点** — 正規化を崩した箇所・論理削除の採否など、判断とその理由（大きければ `adr` へ）

## ルール

- **整合性ルールは DB 制約に置く。** アプリコードのチェックだけに頼らない。UNIQUE・FK・NOT NULL・CHECK で表現できるものは DB に書く（アプリのバグがデータを壊せなくなる）
- **要件にあるデータだけをモデル化する。** 「将来使うかも」のカラム・テーブルは作らない。マイグレーションは後からでも書ける
- **識別子と時刻の方針を最初に決める。** 主キーの形式（連番/UUID）、タイムスタンプの持ち方（UTC統一等）はテーブル間で揃える
- **量を訊く。** 行数の桁（千/百万/十億）でインデックスと分割の判断が変わる。要件に無ければユーザーに確認する

## 完成基準

- [ ] 要件の各ユースケースが読み書きするテーブルが特定できる
- [ ] すべての関係（1:1 / 1:N / N:M）が ER 図に現れている
- [ ] DB 側で強制する制約が列挙されている
- [ ] status ヘッダを付けた
