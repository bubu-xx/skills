# Skills Catalog

開発の各工程・各方法論を細かくスキル化したライブラリ。ウォーターフォールでもアジャイルでも、
どの進め方を選んでも「その作業のスキルがある」状態を目指す。

インストール: `npx skills add bubu-xx/skills -a claude-code`

---

## 設計原理

### スキルには3つの種類がある

工程ではなく「存在の種類」で分類する。種類によって SKILL.md の書き方自体が変わる。

| 種類 | 正体 | 必要な節 | 例 |
|------|------|----------|-----|
| **Mode** | 常駐する振る舞い上書き。on/off があり持続する | 立場 / Persistence / ルール / やめどき / 衝突 | `tdd` `grill` `spike` |
| **Generator** | 一発の成果物生成。入力→出力で完結 | 入力 / 出力の型 / 完成基準 | `adr` `runbook` `user-story` |
| **Orchestrator** | 自分は作業せず、順序とゲートだけ決める指揮者 | ゲート列 / 反復構造 | `waterfall` `agile` `sdd` |

### 背骨は成果物（アーティファクト）

スキル同士は名指しで呼び合わない。**成果物で疎結合にする**。

```
要件メモ ──▶ 設計メモ ──▶ コード＋テスト ──▶ 検証レポート
```

- 各スキルは「どの成果物を作る/更新するか」を宣言する
- Orchestrator のゲートは「どのスキルを実行したか」ではなく「**成果物が存在して基準を満たすか**」だけを見る
- どのスキルで要件メモを作ってもゲートは通る。スキルの増減が指揮者に波及しない

### 方法論依存の「深さ」はゲート側に置く

同じ工程でも、方法論によって詰めの深さが違う（waterfall の要件=凍結、agile の要件=今回のスライス分だけ）。
この差は **L1 スキルの中に持たせず、Orchestrator のゲート基準に置く**。
工程スキルは「どこまで詰めるか」を文脈から受け取るだけで、プロセス非依存を保つ。

### スキルの契約

各 SKILL.md は frontmatter 直下に契約を宣言する:

```
種類: Mode | Generator | Orchestrator
入力: 成果物 or 会話
出力: 成果物 or 振る舞い
やめどき: （Mode のみ）
衝突: （Mode のみ。例: spike ⇔ tdd）
```

### 分割基準

**トリガー文脈が違う、またはルールが衝突する場合のみ別スキルに割る。**
入力が違うだけなら1スキル内で分岐する（例: 叩き台ドキュメントの有無は `grill` 内で分岐。grill-me / grill-with-doc に割らない）。

### 命名原則

業界の定番語があればそれ（`tdd` `ddd` `adr`）。無ければ作業内容を名詞で（`data-model` `test-plan`）。
どちらも無いものだけメタファー可（`grill`）。

### ディレクトリ構成

フラット1階層: `skills/<スキル名>/SKILL.md`。工程カテゴリは下表の索引列であって、ディレクトリではない。

|--------|-----------|---------------|
| 要件メモ | `grill` `user-story` | 設計系・実装系すべて |
| 設計メモ | `ddd` `architecture` `api-first` `data-model` | 実装系 |
| ADR | `adr` | レビュー系 |
| コード＋テスト | `tdd` `ponytail` `spike` | 検証系 |

## カタログ

凡例: **◎核**(背骨・最優先) / **○拡張**(必要時に追加) / **✅既存**(組込み/プラグインを再利用) / **△要検討**(重複ぎみ)

○に入る条件: **固定したいフォーマットか手順を具体的に言えること**。Claude がデフォルトで十分できる作業は除外行き。

### Mode（振る舞いモード）

| スキル | 何を上書きする | 出力 | 工程(索引) | 区分 |
|--------|---------------|------|-----------|------|
| `grill` | 従順さ→詰問。要件確定まで実装しない。叩き台ドキュメントがあれば矛盾・抜けを突く | 要件メモ | 発見 | ✅作成済 |
| `tdd` | Red-Green-Refactor 強制。テスト先行 | コード＋テスト | 実装 | ◎ |
| `ddd` | ユビキタス言語→境界づけ→集約 の順を強制 | 設計メモ | 発見〜設計 | ◎ |
| `spike` | 品質度外視、使い捨て前提で最速に動作検証 | コード(使い捨て) | 実装 | ○ |
| `refactor` | 振る舞い凍結。テストが通る状態を維持したまま改善のみ | コード＋テスト | 実装 | ○ |
| `adversarial-review` | 擁護→攻撃。自分の実装を壊しにいく | 検証レポート | 検証 | ○ |
| `ponytail` | 最小実装 | コード＋テスト | 実装 | ✅プラグイン |
| `bdd` | 振る舞い(Gherkin)から実装 | コード＋テスト | 実装 | △ tddと被る |

Mode の衝突表: `spike` ⇔ `tdd`・`refactor`（品質前提が矛盾） / `grill` は実装系すべてと直列（詰め終わってから）

### Generator（成果物ジェネレータ）

| スキル | 入力 | 出力 | 工程(索引) | 区分 |
|--------|------|------|-----------|------|
| `user-story` | 会話・要件メモ | ストーリー＋受入基準(Given-When-Then) | 発見 | ○ |
| `architecture` | 要件メモ | 全体アーキ・技術選定の設計メモ | 設計 | ○ |
| `api-first` | 要件メモ | API 契約 | 設計 | ○ |
| `data-model` | 要件メモ | スキーマ/ER | 設計 | ○ |
| `nfr` | 要件メモ | 非機能要件表 | 設計 | ○ |
| `adr` | 設計上の判断 | ADR | 設計 | ○ |
| `threat-model` | 設計メモ | 脅威モデル | 設計 | ○ |
| `test-plan` | 要件メモ・設計メモ | テスト観点/計画 | 検証 | ○ |
| `release-checklist` | 検証レポート | リリース前ゲート | リリース | ○ |
| `runbook` | コード・設計メモ | 運用手順書 | 運用 | ○ |
| `observability` | 設計メモ | ログ/メトリクス/トレース設計 | 運用 | ○ |
| `postmortem` | 障害の記録 | ポストモーテム | 運用 | ○ |
| `code-review` | 差分 | 検証レポート | 検証 | ✅組込み |
| `security-review` | 差分 | 検証レポート | 検証 | ✅組込み |
| `verify` | コード | 動作確認 | 検証 | ✅組込み |

### Orchestrator（指揮者）

| スキル | 反復構造 | ゲート基準 | 区分 |
|--------|----------|-----------|------|
| `waterfall` | 一直線・一巡 | 各成果物を**凍結**してから次へ | ◎ |
| `agile` | 薄いスライスで多巡 | 各成果物は**今回のスライス分**で通す。凍結しない | ◎ |
| `sdd` | 仕様先行 | 仕様を正典として凍結→実装・検証は仕様との一致で判定 | ◎ |
| `lean-mvp` | 仮説→検証ループ | 価値仮説の検証可否 | ○ |
| `scrum` / `kanban` | セレモニー/フロー込み | — | △ agileと被る |

## 除外（意図的に載せていない）

- `brainstorm` `problem-framing` `value-hypothesis` `impact-mapping` `retro` `refinement` … Claude がデフォルトで十分。固定すべき独自フォーマットが出てきたら Generator として復活させる
- `c4-diagram` … `ddd` / `architecture` に内包
- `pair-programming` … 常時そうしている
- `event-storming` … `ddd` の発見フェーズに内包（DDD が発見〜設計をまたぐ Mode である、が正しい形）
- `grill-me` / `grill-with-doc` … `grill` に統合（分割基準を満たさない）

---

## ビルド戦略

全部を一気に作らない。使って検証していないスキルは腐る。

1. **背骨から**: `grill`(済) → `tdd` `ddd` `sdd` `waterfall` `agile`
2. **最初の Orchestrator（waterfall）を作る時に検証すること**: スキルからスキル/成果物への連携が実際に機能するか。ゲートが成果物だけを見る設計が成立するか。**ここがカタログ全体の前提なので最優先で潰す**
3. 残りは使う時に雛形から複製（Mode は `skills/grill`、Generator/Orchestrator は最初に作ったものが雛形）
4. △要検討 は核を作ってから被り具合を見て判断
