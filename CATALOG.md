# Skills Catalog

開発の各工程を細かくスキル化したライブラリ。ウォーターフォールでもアジャイルでも、
どのプロセスを踏んでも「その工程のスキルがある」状態を目指す。

インストール: `npx skills add bubu-xx/skills -a claude-code`

## 設計原理

各スキルは「そのモードの間だけ Claude の振る舞いを上書きするシステムプロンプト」。
中身は **立場 → ルール → 出力の型 → 相性** の骨格で統一する（`skills/grill-me` が雛形）。

スキルは2層に分ける。層の境界は「方法論が何工程をまたぐか」で決める:

- **L1 工程スキル** — 1工程に収まる作業。プロセス非依存の部品。TDD(実装)・DDD(設計)のように、1工程に収まる方法論もここ。
- **L2 方法論スキル** — 複数工程をまたぐ進め方。自分では作業せず、L1 を「どの順で・どう反復するか」だけ指揮する。waterfall / agile / SDD など。

これで「どのプロセスでも工程が埋まってる」が成立する。部品は共通、指揮者だけ差し替える。

## ディレクトリ構成

```
skills/<カテゴリ>/<スキル名>/SKILL.md
```

カテゴリ = 下のライフサイクル工程名。`npx skills` はこの2階層カタログ構成に対応済み。

## 凡例

- **◎核** … 背骨。最優先で作る（デフォルトと最も違う振る舞い）
- **○拡張** … 必要になったら雛形から複製
- **✅既存** … 組込み/プラグインで既にある。作らず再利用
- **△要検討** … 他スキルと重複ぎみ。単独で作るか「相性」で誘導するか判断

## L1 工程スキル

### planning（企画）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `problem-framing` | 解を出す前に「解くべき課題」を定義する | ○ |
| `value-hypothesis` | 誰にどんな価値か、の仮説を立てる（リーン） | ○ |
| `impact-mapping` | ゴールから逆算して施策を洗い出す | ○ |

### discovery（発見/要件）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `grill-me` | 詰問で曖昧要件を確定（対話のみ） | ✅作成済 |
| `grill-with-doc` | 既存ドキュメント基準で矛盾・抜けを突く | ◎ |
| `user-story` | ユーザーストーリー＋受入基準(Given-When-Then) | ○ |
| `event-storming` | ドメインイベントを洗い出す（DDD発見） | △ |

### design（分析/設計）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `ddd` | ユビキタス言語→境界づけ→集約 の順で設計 | ◎ |
| `architecture` | 全体アーキ・技術選定 | ○ |
| `api-first` | 実装前に API 契約を確定 | ○ |
| `data-model` | スキーマ/ER 設計 | ○ |
| `nfr` | 非機能要件（性能/可用性/運用）を詰める | ○ |
| `adr` | 設計判断を ADR として記録 | ○ |
| `threat-model` | セキュリティ設計 | ○ |

### implementation（実装）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `tdd` | Red-Green-Refactor、テスト先行 | ◎ |
| `bdd` | 振る舞い(Gherkin)から実装 | △ |
| `ponytail` | 最小実装 | ✅プラグイン |
| `spike` | 使い捨て前提で最速に動作検証 | ○ |
| `refactor` | 振る舞いを保ったまま改善 | ○ |

### verification（検証）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `code-review` | 差分レビュー | ✅組込み |
| `security-review` | 脆弱性レビュー | ✅組込み |
| `verify` | 実際に動かして確認 | ✅組込み |
| `adversarial-review` | 自分の実装を壊しにいく | ○ |
| `test-plan` | テスト観点/計画 | ○ |

### delivery（リリース/運用）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `release-checklist` | 出す前の確認ゲート | ○ |
| `runbook` | 運用手順書 | ○ |
| `observability` | ログ/メトリクス/トレース設計 | ○ |
| `postmortem` | 障害振り返り | ○ |

### retrospective（振り返り）
| スキル | 何をする | 区分 |
|--------|----------|------|
| `retro` | スプリント振り返り | ○ |
| `refinement` | バックログ整理 | ○ |

## L2 方法論スキル（指揮者）

| スキル | 何をする | 区分 |
|--------|----------|------|
| `waterfall` | L1 を厳格な順＋工程ゲートで一直線に | ◎ |
| `sdd` | 仕様を正典として先に固め、それ駆動で実装・検証 | ◎ |
| `agile-sprint` | L1 を短い反復ループで回す | ○ |
| `lean-mvp` | 最小価値仮説→検証で回す | ○ |
| `scrum` / `kanban` | セレモニー/フロー込みの進め方 | △ |

---

## 除外（意図的に載せていない）

- `brainstorm` … Claude がデフォルトで十分
- `c4-diagram` … `ddd` / `architecture` に内包
- `pair-programming` … 常時そうしている

## ビルド戦略

全部を一気に作らない。使って検証していないスキルは腐る。

1. **背骨から** … `grill-with-doc` `ddd` `tdd` `sdd` ＋ 指揮者雛形 `waterfall`
2. **残りは使う時に** … 雛形（`skills/grill-me`）から複製して中身を差し替える
3. `△要検討` は、核を作ってから被り具合を見て判断
