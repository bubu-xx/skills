# Skills Catalog

開発の各工程・各方法論を細かくスキル化したライブラリ。ウォーターフォールでもアジャイルでも、
どの進め方を選んでも「その作業のスキルがある」状態を目指す。

インストール: `npx skills add bubu-xx/skills -a claude-code`

---

## 設計原理

### スキルには3つの種類がある

工程ではなく「存在の種類」で分類する。種類によって SKILL.md の書き方自体が変わる。

| 種類 | 正体 | 必要な節 | 例 |
|------|------|----------|-----|
| **Mode** | 常駐する振る舞い上書き。動作中は振る舞いを変え、終了時に成果物を残す | 契約 / 立場 / Persistence / ルール / やめどき | `tdd` `grill` `spike` |
| **Generator** | 一発の成果物生成。入力→出力で完結 | 契約 / 出力の型 / 完成基準 | `adr` `runbook` `user-story` |
| **Orchestrator** | 自分は作業せず、順序とゲートだけ決める指揮者 | 契約 / ゲート列 / 反復構造 | `waterfall` `agile` `sdd` |

### 背骨は成果物（アーティファクト）

スキル同士は名指しで呼び合わない。**成果物で疎結合にする**。

成果物は5つの**型**に統一する。Generator が作る個別ドキュメント（API契約、ER図、ADR…）は型の**変種**であり、ゲートは型しか見ない。

| 型 | 変種の例 | 置き場所（規約） |
|----|----------|-----------------|
| 要件 | 要件メモ、ユーザーストーリー | `docs/<機能名>/requirements.md` |
| 設計 | 設計メモ、API契約、ER、ADR、脅威モデル、非機能要件表 | `docs/<機能名>/design.md`（変種は同ディレクトリに追加ファイル） |
| コード＋テスト | 実装、スパイク | リポジトリ本体 |
| 検証 | レビュー結果、テスト計画、動作確認記録 | `docs/<機能名>/verification.md` |
| 運用 | runbook、observability設計、postmortem | `docs/<機能名>/ops/` または `docs/ops/` |

### 成果物ヘッダ規約 — ゲートを機械化する

ドキュメント型の成果物は frontmatter に状態を持つ:

```yaml
---
type: 要件 | 設計 | 検証 | 運用
feature: <機能名>
status: draft | slice-ok | frozen
---
```

- `draft` … 作業中。ゲートは通らない
- `slice-ok` … 今回のスライスを作るには足りる（agile のゲート基準）
- `frozen` … 凍結。後戻りしない前提（waterfall / sdd のゲート基準）

**ゲート＝「対象の型の成果物が存在し、status が基準以上」**。会話の記憶ではなくファイルを見るので、機械的にチェックできる。

### Orchestrator の状態＝ディスク上の成果物

指揮者は「今どの工程か」を会話コンテキストで覚えない。**どの成果物がどの status で存在するか**から現在地を再導出する。だから:

- セッションをまたいで再開できる
- コンテキストが圧縮されても壊れない
- 下流スキルは深さ（status 目標）を会話でなく成果物ヘッダから読む

これが成果物を背骨にする真の理由。

### 方法論依存の「深さ」はゲート側に置く

同じ工程スキルでも、方法論によって詰めの深さが違う（waterfall の要件=frozen、agile の要件=slice-ok）。
この差は**工程スキルの中に持たせず、Orchestrator が指定する status 目標**として渡す。
工程スキルは目標 status に達したら止まるだけで、プロセス非依存を保つ。

### スキルの契約

各 SKILL.md は frontmatter 直下に契約を宣言する:

```
種類: Mode | Generator | Orchestrator
入力: 成果物の型 or 会話
出力: 成果物の型（Mode は「動作中の振る舞い＋終了時に残す成果物」）
やめどき: （Mode のみ）
衝突: 共存できない Mode（例: spike ⇔ tdd）
順序: 先行が必要な関係（例: grill → 実装系）
```

**衝突**（共存不可）と**順序**（先行必須）は別の関係。混ぜない。

### 分割基準

**トリガー文脈が違う、またはルールが衝突する場合のみ別スキルに割る。**
入力が違うだけなら1スキル内で分岐する（例: 叩き台ドキュメントの有無は `grill` 内で分岐。grill-me / grill-with-doc に割らない）。

### 命名原則

業界の定番語があればそれ（`tdd` `ddd` `adr`）。無ければ作業内容を名詞で（`data-model` `test-plan`）。
どちらも無いものだけメタファー可（`grill`）。

### ディレクトリ構成

種類で2階層: `skills/<種類>/<スキル名>/SKILL.md`（種類 = `mode` | `generator` | `orchestrator`）。
工程カテゴリは下表の索引列であって、ディレクトリではない。フラット配置と混在させない（浅い SKILL.md が優先され発見が壊れる）。

### 起動方式

- Mode / Generator は自動起動可（description のトリガー語で発動）
- **Orchestrator は明示起動のみ**（`disable-model-invocation: true`）。「アジャイルの話をしただけで /agile が走る」誤発火を防ぐ

### 各スキルの必須節

frontmatter（name / description）、契約ブロック、**防ぐ失敗**（このスキルが無いとき何が起きるかの1〜2行＋具体例1個）。
`python scripts/validate_skills.py` で機械チェックできる。

## カタログ

凡例: **◎核**(背骨) / **○拡張** / **✅既存**(再利用) / **△要検討**(重複ぎみ・未実装)

- **◎・○ はすべて実装済み**（`skills/` 配下に SKILL.md がある）。△ のみ保留
- ○に入る条件: **固定したいフォーマットか手順を具体的に言えること**。Claude がデフォルトで十分できる作業は除外行き
- ✅組込み は Claude Code 本体の機能を指す。**利用者の環境に依存する**ので、無い環境では ○ 扱いで自作候補になる

### Mode（振る舞いモード）

| スキル | 何を上書きする | 残す成果物 | 工程(索引) | 区分 |
|--------|---------------|-----------|-----------|------|
| `grill` | 従順さ→詰問。要件確定まで実装しない。叩き台ドキュメントがあれば矛盾・抜けを突く | 要件 | 発見 | ◎ |
| `tdd` | Red-Green-Refactor 強制。テスト先行 | コード＋テスト | 実装 | ◎ |
| `ddd` | ユビキタス言語→境界づけ→集約 の順を強制 | 設計 | 発見〜設計 | ◎ |
| `spike` | 品質度外視、使い捨て前提で最速に動作検証 | コード(使い捨て) | 実装 | ○ |
| `refactor` | 振る舞い凍結。テストが通る状態を維持したまま改善のみ | コード＋テスト | 実装 | ○ |
| `adversarial-review` | 擁護→攻撃。自分の実装を壊しにいく | 検証 | 検証 | ○ |
| `ponytail` | 最小実装 | コード＋テスト | 実装 | ✅プラグイン |
| `bdd` | 振る舞い(Gherkin)から実装 | コード＋テスト | 実装 | △ tddと被る |

関係表:
- **衝突**: `spike` ⇔ `tdd`、`spike` ⇔ `refactor`（品質前提が矛盾）
- **順序**: `grill` → 設計・実装系すべて（詰め終わってから）

解説:

- **`grill`** — 曖昧な要望に対し、賛同や実装から入らず質問で穴を突く。要件が固まるまでコードを書かない。メリット: 作ってから「思ってたのと違う」となる手戻りを最初に潰せる。
- **`tdd`**（テスト駆動開発） — コードを書く前に「正しければ通るはずのテスト」を先に書く。①失敗するテスト(Red)→②通る最小実装(Green)→③動きを変えず整理(Refactor)の3拍子を繰り返す。メリット: 「何ができたら完成か」がテストとして先に確定するので、作りすぎ・仕様漏れが減り、後から安心して変更できる。
- **`ddd`**（ドメイン駆動設計） — 業務の言葉をそのままコード構造にする設計法。用語集（ユビキタス言語）→業務の意味の切れ目で分割（境界づけ）→データのまとまりと整合性ルール（集約）の順を強制。メリット: 「営業の言う『キャンセル』とシステムの『キャンセル』が違う」類のズレが設計段階で潰れる。複雑な業務システム向け。単純なツールには大げさ。
- **`spike`**（スパイク） — 「これ技術的に可能なの?」を確かめるためだけの使い捨てコード。品質・テストを全部無視して最短で動くか検証し、**確認できたら捨てる**。メリット: 不確かな技術に本気の実装コストを払う前に、安く答えが出る。
- **`refactor`**（リファクタリング） — 外から見た動きを一切変えずに中身だけ整理する。機能追加・バグ修正は禁止、テストが通り続けることを保つ。メリット: 機能変更と整理を混ぜて「直してたら別の場所が壊れた」となる事故を防ぐ。
- **`adversarial-review`**（敵対的レビュー） — 自分の実装を「どこかが壊れているはず」という攻撃者目線で見直し、壊す方法だけを探す。メリット: 良いところ探しに流れがちなセルフレビューより欠陥検出率が上がる。
- **`ponytail`** — 動く最小限だけを書く。標準ライブラリ優先、投機的な抽象化禁止。メリット: 書かないコードはバグらない・保守もいらない。
- **`bdd`**（振る舞い駆動開発） — テストを「Given(〜の状態で) When(〜したら) Then(〜になる)」の文形式で書く TDD の親戚。メリット: 非エンジニアともテストを共有できる。TDD と役割が被るため△。

### Generator（成果物ジェネレータ）

| スキル | 入力 | 出力（型の変種） | 工程(索引) | 区分 |
|--------|------|-----------------|-----------|------|
| `user-story` | 会話・要件 | 要件: ストーリー＋受入基準(Given-When-Then) | 発見 | ○ |
| `architecture` | 要件 | 設計: 全体アーキ・技術選定 | 設計 | ○ |
| `api-first` | 要件 | 設計: API 契約 | 設計 | ○ |
| `data-model` | 要件 | 設計: スキーマ/ER | 設計 | ○ |
| `nfr` | 要件 | 設計: 非機能要件表 | 設計 | ○ |
| `adr` | 設計中の判断 | 設計: ADR | 設計 | ○ |
| `threat-model` | 設計 | 設計: 脅威モデル | 設計 | ○ |
| `test-plan` | 要件・設計 | 検証: テスト観点/計画 | 検証 | ○ |
| `release-checklist` | 検証 | 検証: リリース前ゲート | リリース | ○ |
| `runbook` | コード・設計 | 運用: 手順書 | 運用 | ○ |
| `observability` | 設計 | 運用: ログ/メトリクス/トレース設計 | 運用 | ○ |
| `postmortem` | 障害の記録 | 運用: ポストモーテム | 運用 | ○ |
| `code-review` | 差分 | 検証: レビュー結果 | 検証 | ✅組込み |
| `security-review` | 差分 | 検証: レビュー結果 | 検証 | ✅組込み |
| `verify` | コード | 検証: 動作確認 | 検証 | ✅組込み |

解説:

- **`user-story`**（ユーザーストーリー） — 要件を「〈誰〉として〈何〉がしたい、なぜなら〈理由〉」の型で書き、「これが満たされたら完成」という受入基準を付ける。アジャイルの定番。メリット: 機能一覧ではなく利用者の目的で要件が整理され、優先順位が付けやすい。
- **`architecture`** — 全体の構成と技術選定（DB は何、通信はどうする等）を決める文書。メリット: 個別実装に入る前に全体の整合が取れる。
- **`api-first`**（API 設計先行） — フロントとバックの通信の取り決め（エンドポイント、リクエスト/レスポンスの形）を実装前に確定させる。メリット: 契約さえ決まればフロントとバックを並行開発できる。
- **`data-model`** — テーブル構造とデータ同士の関係（ER 図）を設計する。メリット: データ構造の間違いは後から直すのが最も高くつくので、先に固める価値が大きい。
- **`nfr`**（非機能要件） — 「何ができるか」以外の要件。応答速度、同時利用者数、障害復旧時間、セキュリティ水準など。メリット: 忘れられがちで後から直すのが一番高くつく類の要件を、先に表に出せる。
- **`adr`**（アーキテクチャ決定記録） — 「なぜ PostgreSQL にしたか」「なぜこのライブラリか」という設計判断の理由を短い文書で残す。メリット: 半年後に「なんでこうしたんだっけ」と同じ議論を繰り返さない。
- **`threat-model`**（脅威モデリング） — 「攻撃者ならどこを狙うか」を設計段階で洗い出す。認証の穴、データ漏洩経路など。メリット: セキュリティ欠陥は実装後の修正より設計時の防止が桁違いに安い。
- **`test-plan`** — 何をどうテストするかの観点表。正常系・異常系・境界値など。メリット: 場当たり的なテストで抜ける観点（異常系・境界値）を網羅できる。
- **`release-checklist`** — 世に出す前の最終確認リスト（バックアップ、ロールバック手順等）。メリット: リリース直前の緊張状態でも確認漏れが出ない。
- **`runbook`**（運用手順書） — 「サーバーが落ちたらこの順で対応」という運用時のマニュアル。メリット: 障害時に調べながら対応する羽目にならない。
- **`observability`**（可観測性） — 動いているシステムの中身が見えるようにする設計。ログに何を書くか、どの数値を監視するか。メリット: 障害時に「何が起きているか分からない」を防ぐ。
- **`postmortem`**（ポストモーテム） — 障害後の振り返り文書。何が起きた・なぜ・再発防止。犯人探しをしないのが流儀。メリット: 同じ障害を二度起こさない仕組みになる。
- **`code-review` / `security-review` / `verify`**（組込み） — それぞれ差分のバグレビュー、脆弱性レビュー、実際に動かしての動作確認。Claude Code 本体の機能。

### Orchestrator（指揮者）

| スキル | 反復構造 | ゲート基準（status） | 区分 |
|--------|----------|---------------------|------|
| `waterfall` | 一直線・一巡 | 各型を `frozen` にしてから次へ | ◎ |
| `agile` | 薄いスライスで多巡 | 各型は `slice-ok` で通す。凍結しない | ◎ |
| `sdd` | 仕様先行 | 要件・設計を `frozen`（正典）→実装・検証は仕様との一致で判定 | ◎ |
| `lean-mvp` | 仮説→検証ループ | 価値仮説の検証可否 | ○ |
| `scrum` / `kanban` | セレモニー/フロー込み | — | △ agileと被る |

解説:

- **`waterfall`**（ウォーターフォール） — 要件→設計→実装→検証を一方通行で一巡する。各段階を完全に固めて（frozen）から次へ進む。メリット: 要件が最初から明確で後戻りが高くつく開発では、工程ごとに確実に固める方が速い。
- **`agile`**（アジャイル） — 全体を薄く輪切りにして「小さく作って動かして直す」を何周も回す。各段階は今回の分だけ決めて進む（slice-ok）。メリット: 作りながら要件が見えてくる開発で、間違った方向に大きく進む前に軌道修正できる。
- **`sdd`**（仕様駆動開発） — まず仕様書を完全に固めて「正典」とし、実装もテストも仕様書との一致で判定する。変更したくなったら先に仕様書を直す。メリット: 実装と仕様のズレが構造的に起きない。AI に実装を任せる開発と相性が良い（仕様書がプロンプトの役割を果たす）。
- **`lean-mvp`**（リーン MVP） — 「そもそも欲しい人いるの?」を最小限の実装で検証してから本格投資する。作る前に価値仮説を立て、検証で回す。メリット: 誰も使わないものを作り込む無駄を防ぐ。
- **`scrum` / `kanban`** — アジャイルの具体的な運営方式。スクラムは「一定期間の区切り＋定例イベント」、カンバンは「タスクを列に流して同時作業数を制限」。個人開発では `agile` との差が薄いため△。

## 除外（意図的に載せていない）

- `brainstorm` `problem-framing` `value-hypothesis` `impact-mapping` `retro` `refinement` … Claude がデフォルトで十分。固定すべき独自フォーマットが出てきたら Generator として復活させる
- `c4-diagram` … `ddd` / `architecture` に内包
- `pair-programming` … 常時そうしている
- `event-storming` … `ddd` の発見フェーズに内包（DDD が発見〜設計をまたぐ Mode である、が正しい形）
- `grill-me` / `grill-with-doc` … `grill` に統合（分割基準を満たさない）

## ビルド戦略

◎・○ は実装済み。次のフェーズは**実運用での検証**:

1. **最初の実戦で検証すること**（waterfall か agile を1機能に通しで使う）:
   - 成果物ヘッダ（status）の読み書きが実際に機能するか
   - ゲートが「ファイルの存在＋status」だけで判定できるか
   - セッションをまたいだ再開が成果物から再導出できるか
2. 使って問題が出たスキルから直す。**使っていないスキルの推敲はしない**
3. △要検討（`bdd` `scrum`/`kanban`）は運用で必要性が実証されたら実装
4. 新スキルの雛形: Mode は `skills/mode/grill`、Generator は `skills/generator/user-story`、Orchestrator は `skills/orchestrator/waterfall`。作ったら `python scripts/validate_skills.py` を通す
